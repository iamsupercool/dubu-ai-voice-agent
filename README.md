# Dubu Voice

아동과 생성형 AI가 1:1로 실시간 대화할 수 있는 웹 기반 음성 대화 서비스입니다. 사용자는 브라우저에서 텍스트 또는 음성으로 메시지를 입력하고, 서버는 WebSocket으로 Ollama 기반 AI 응답을 스트리밍합니다. 대화 내용은 사용자별 JSON 파일로 저장되어 대화 목록 조회, 상세 조회, 삭제, 이어서 대화가 가능합니다.

## 1. 사용 방법

### 사전 준비

- Node.js 20 이상
- npm
- Chrome 브라우저 필수
  - 음성 입력은 브라우저 Web Speech API를 사용하므로 Chrome 환경에서 가장 안정적으로 동작합니다.
- Ollama 설치
  - 로컬 LLM 서버로 사용합니다.

### Ollama 모델 준비

현재 서버는 `nest-api/src/services/ai.service.ts`에서 `gemma3` 모델을 사용합니다.
#### AI Model 설치  
##### Window
https://ollama.com/
다운로드 설치

##### Mac
```bash
brew install ollama
```
#### ollama 설정
1. ollama 실행
```bash
ollama serve
```
2. AI 모델 다운로드 (최초 설치 후 실행) > 다운로드 후 종료
```bash
ollama pull gemma3
```

#### Ollama 서버 실행 (병렬처리를 위한 옵션을 포함)

```bash
npm run start:ollama
```

또는 직접 실행할 수도 있습니다.

```bash
OLLAMA_NUM_PARALLEL=4 OLLAMA_MAX_LOADED_MODELS=2 OLLAMA_KEEP_ALIVE=30m ollama serve
```

### 환경 변수 설정
env는 편의를 위해 프로젝트 내에 포함되어 있습니다.

### 의존성 설치

루트에서 전체 의존성을 설치합니다.

```bash
npm run install:all
```

### 개발 서버 실행

Ollama가 실행 중인 상태에서 프론트엔드와 백엔드를 함께 실행합니다.

```bash
npm run dev:all
```

접속 주소:

- 웹 클라이언트: http://localhost:3000
- API 서버: http://localhost:3001
- Swagger 문서: http://localhost:3001/api-docs

### 기본 사용 흐름

1. http://localhost:3000 에 접속합니다.
2. 회원가입 후 로그인합니다.
3. 채팅 화면에서 텍스트 또는 음성으로 첫 질문을 입력합니다.
4. 첫 질문이 전송되면 새 대화가 생성되고, AI 응답이 실시간 말풍선으로 스트리밍됩니다.
5. 좌측 대화 목록에서 이전 대화를 선택해 이어서 대화하거나 삭제할 수 있습니다.


## 2. 주요 기술 스택

### 공통
- concurrently
  - mono repo 환경에서 두개의 서비스를 동시에 실행하기 위해 사용

### Frontend

- Next.js 14 Page Router
  - 과제 범위가 명확한 페이지 중심 앱이므로 `pages/index.tsx`, `pages/chat.tsx` 구조로 단순하게 구성했습니다.
- Tailwind CSS, shadcn/ui, Radix UI
  - 빠르게 일관된 UI를 만들고, 채팅 입력/목록/다이얼로그 같은 반복 UI의 접근성을 확보하기 위해 사용했습니다.
  - AI가 쉽게 이해하고 만들 수 있는 조합으로 선택하였습니다.
- TanStack Query
  - 대화 목록을 10초 간격으로 자동 갱신하고, 삭제 후 목록을 자연스럽게 다시 불러오기 위해 사용했습니다.
  - 대화 목록은 빠르게 갱신하여 UX가 완성도 있길 원했습니다.
- Socket.io Client
  - AI 응답 토큰 스트리밍, 대화 재개, 응답 중단 이벤트를 양방향으로 처리하기 위해 사용했습니다.
- Web Speech API
  - 별도 STT/TTS 외부 서비스를 붙이지 않고 브라우저 내장 음성 인식과 음성 합성을 사용해 로컬 실행성과 응답 지연을 우선했습니다.

### Backend

- Nest.js
  - 인증, 대화 기록, WebSocket 게이트웨이를 모듈 단위로 나누기 쉽고, 프레임워크를 통해 서버 구조를 명확하게 유지할 수 있었습니다.
- Socket.io
  - 탭 단위 연결 상태와 실시간 AI 스트리밍 이벤트를 다루기 위해 사용했습니다.
- JWT, Passport
  - REST API와 WebSocket 연결 모두 동일한 Bearer 토큰 인증 흐름을 사용합니다.
  - front-end, back-end 분산 환경에서 인증을 만들고, token에 있는 정보로 사용자 인증처리를 하기 위해 사용했습니다.
- JSON 파일 DB
  - 과제 환경에서 별도 DB 설치 없이 실행할 수 있도록 파일 기반으로 선택하였습니다.
  - 복잡한 과정보다 AI 음성 에이전트 서버의 흐름에 집중했습니다.
- Ollama
  - 무료 로컬 LLM 서버를 사용했습니다.

## 3. 구현 범위 및 미구현 사항

### 구현한 범위
#### ① AI 음성 에이전트 서버
- `VoiceGateway`에서 WebSocket 연결, 인증, 메시지 수신, AI 응답 스트리밍, 응답 중단을 처리합니다.
- `AiService`에서 Ollama `/api/chat`의 `stream: true` 응답을 토큰 단위로 읽어 클라이언트에 전달합니다.
#### ② 웹 클라이언트
- 로그인/회원가입, 채팅 화면, 대화 목록 사이드바, 텍스트 입력, 음성 입력, 응답 중단 UI를 구현했습니다.
#### ③ 대화 로그 표시
- 사용자 말풍선과 AI 말풍선을 구분하고, AI 응답은 `ai:stream:token` 이벤트를 받아 실시간으로 갱신합니다.
- 음성 입력 중인 미리보기 문장도 채팅 영역의 말풍선으로 표시합니다.
- 30개 단위 커서 페이지네이션
#### ④ 사용자 인증 및 대화 기록 조회  
- 로그인 후 토큰은 `sessionStorage`에 저장합니다.
- 대화 목록 조회, 상세 조회, 삭제, 이어서 대화를 구현했습니다.
#### ⑤ 다중 사용자 동시 접속
- 소켓 연결은 `socket.id` 기준으로 세션을 분리합니다.
- `MessageQueueService`로 같은 탭에서는 메시지 순서를 보장하고, 다른 탭/사용자는 병렬 처리되도록 구성했습니다.

### 미구현 또는 제한 사항

- 실제 운영용 DB는 사용하지 않았습니다. JSON 파일 저장 방식은 로컬 과제 실행에 맞춰 기능 구현에 중점을 뒀습니다.
- 음성 인식은 Web Speech API 의존성이 있어 브라우저별 품질 차이가 있습니다. Chrome 외 환경에서는 음성 기능이 제한될 수 있습니다.
- LLM 모델 로딩 시간과 응답 속도는 로컬 머신 성능 및 Ollama 모델 상태에 영향을 받습니다.

## 4. 설계/구현 중 중점을 둔 포인트

- Claude, Codex(무료) 버전으로 제작하였습니다.
- UI는 ChatGPT를 벤치마킹하였습니다.

### 첫 응답 지연 줄이기

- AI 응답을 완성된 문장 단위로 기다리지 않고 Ollama의 `stream: true` 응답을 토큰 단위로 클라이언트에 즉시 전달했습니다. 클라이언트는 스트리밍 말풍선을 바로 생성하고 토큰이 도착할 때마다 텍스트를 누적해, 사용자가 응답이 시작되었다는 느낌을 빠르게 받을 수 있게 했습니다.

### 탭 독립성과 세션 분리
- 다중 사용자 동시 접속 요구사항

### 대화 저장 시점

새 대화는 첫 질문이 들어올 때 생성합니다. 빈 대화가 쌓이지 않게 하기 위한 결정입니다. 이후 사용자 메시지와 AI 메시지는 응답이 오갈 때마다 JSON 파일에 즉시 추가해, 새로고침이나 재접속 후에도 대화 기록을 복구할 수 있게 했습니다.

### 음성 UX

- 사용자의 음성을 놓치지 않도록 마이크가 활성화 된 후 음성 입력이 시작. 
- 마이크 볼륨을 분석해 무음 구간을 감지하고 자동 종료. 
- 인식 중인 문장은 입력창 내부 텍스트가 아니라 채팅 영역의 사용자 말풍선으로 표시해, 실제 대화 흐름 안에서 사용자가 자신의 발화를 확인. 
- AI 음성 출력은 `SentenceDetector`와 TTS 큐를 통해 문장 단위로 빠르게 재생되도록 구성. 아이의 환경에 맞춰 말하기 속도를 0.9로 설정

### 연결 상태와 복구

연결이 끊겼을 때, 사용자 안내와 재연결에 대한 처리

## 5. 추후 고려 사항

- WebSocket과 WebRTC 중 적절한 스펙이 무엇인지 고민할 것
- JSON 파일 저장소를 적절한 Database 구조와 설계를 갖출 것
- AI의 페르소나 강화 및 지식 추가
- ChatGPT의 음성 대화 모드와 같이 toggle UI가 아닌 지속적인 음성 세션으로 대화하는 UX로 전환
- 음성 인식의 감도를 사용자 설정으로 전환
